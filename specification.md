# SDD Navigator Dashboard – Спецификация

## 1. Контекст
Веб-приложение для визуализации и анализа метрик покрытия спецификаций в системе разработки.
Цель — дать разработчикам и тест-менеджерам единую точку входа для оценки полноты тестирования и документирования компонентов.

## 2. Цели
1) Пользователь видит общую метрику покрытия спецификаций (%) и трендовую линию.
2) Пользователь просматривает список модулей с индивидуальным покрытием каждого.
3) Пользователь кликает на модуль и видит список спецификаций с их статусом (covered/not covered).
4) Пользователь может фильтровать модули по статусу (excellent/good/warning/critical) и искать по названию.
5) Пользователь видит детали: дата последнего обновления, автор, статус проверки.

## 3. Ограничения (ненужные задачи)
- Редактирование спецификаций и статусов через UI (только для просмотра).
- Интеграция с внешними системами (Jira, TestRail и т.д.) — в MVP используются моковые данные.
- Экспорт отчётов (PDF, Excel).
- Историческое сравнение версий спецификаций (только текущее состояние).
- Углублённый анализ трендов и прогнозы.

## 4. Пользовательские сценарии

**S1. Главный дашборд:**
- Карточка с общей метрикой: % покрытия (например, 80.7%)
- Показатели: кол-во модулей (5), спецификаций (60), покрыто (48)
- График-линия: история покрытия за последние 7 дней

**S2. Список модулей:**
- Таблица/карточки: Название, Покрытие (%), Статус (excellent/good/warning/critical)
- Цветовая кодировка: зелёный (≥95%), светло-зелёный (80%-94%), жёлтый (60–79%), красный (<60%)
- Сортировка по покрытию (убыванию)

**S3. Детали модуля (после клика):**
- Название модуля и его метрики
- Список спецификаций (API endpoints, функции и т.д.)
- Столбцы: название, статус (covered/not covered), дата последней проверки
- Кнопка "Назад" в список модулей

**S4. Поиск и фильтр:**
- Поле ввода: поиск по названию модуля
- Чекбоксы: только "excellent", только "good", только "warning", только "critical"
- Результаты обновляются в реальном времени

**S5. Загрузка данных:**
- При открытии дашборда данные загружаются с API Routes (`/api/metrics`, `/api/modules`, `/api/specifications`)
- Индикатор загрузки (spinner) во время запроса

## 5. Бизнес-правила
- **Покрытие = (Covered Specs / Total Specs) × 100%** округляется до одного знака.
- **Статус модуля** определяется по его покрытию: excellent ≥95%, good 80–94%, warning 60–79%, critical <60%.
- **Дата обновления** — дата последней проверки любой спецификации в модуле.
- Данные обновляются при каждой загрузке страницы (GET запрос к API).

## 6. Архитектурные решения
- **Фронтенд:** Next.js, React, Tailwind CSS / Material-UI для стилей
- **API:** Next.js API Routes (`pages/api/`)
  - `GET /api/metrics` — общие метрики (overall coverage, trend, counts)
  - `GET /api/modules` — список всех модулей
  - `GET /api/modules/[id]` — детали модуля + его спецификации
- **Данные:** моковые данные, генерируются на лету (Faker или static JSON)
- **Состояние:** React hooks (useState, useEffect)

## 7. Критерии приемки
- [ ] Все сценарии S1–S5 воспроизводимы через интерфейс
- [ ] Поиск и фильтр работают без перезагрузки страницы
- [ ] API Routes возвращают корректный JSON и HTTP статусы
- [ ] Страница отображается без ошибок в консоли (prod mode)
- [ ] Минимум 2 интеграционных теста: загрузка дашборда, клик на модуль
- [ ] README содержит: как запустить, структуру API, примеры запросов
- [ ] Код проходит линтер (ESLint)
